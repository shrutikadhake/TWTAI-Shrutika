The Prompt → Skill → Agent Spectrum

Simple ─────────────────────────────────── Complex

One-off Prompt      Skill (Reusable)      Agent (Autonomous)

"Fix this typo"     /check-api-doc        Full automation

Low effort          Medium effort         High capability

Inconsistent        Consistent            Most powerful

Your goal today: move one task to a reusable skill.

The RTCCO Framework

ElementWhat It Does
RoleSets the AI's perspective and expertise
TaskWhat the AI should do
ContextBackground the AI needs
ConstraintsBoundaries and rules
Output formatHow the result should look
Most beginners include Role and Task.
The differentiator is Context, Constraints, and Output.
5 / 12

Prompt Quality Comparison

Weak (no RTCCO):

Review this file.

Partial (Role + Task only):

You are a technical editor.
Review this API doc for completeness.

Strong (full RTCCO):

You are a senior technical editor with 10 years in API documentation.
Review the API endpoint doc at $ARGUMENTS for completeness.
Context: Readers are junior developers. We follow Google Dev Style Guide.
Constraints: Do NOT modify code. Flag issues in prose only.
Output: Markdown table — Section | Status | Finding | Fix

Anatomy of a Claude Skill File

.claude/
  commands/
    check-api-doc.md    ← your skill lives here
You are a [ROLE].                         ← R

Review the file at: $ARGUMENTS            ← T + input

Context: [who reads it, style guide]      ← C

Check for:
1. [Criterion 1]
2. [Criterion 2]                          ← detailed task

Constraints: [what NOT to do]             ← C

Output: [exact format description]        ← O
7 / 12

The $ARGUMENTS Variable

Review the documentation at: $ARGUMENTS

When you run:

/check-api-doc path/to/my-api.md

$ARGUMENTS becomes path/to/my-api.md

This is what makes a skill reusable — it works on any file.

Build Your First Skill (Workshop)

Step 1: Create the folder

mkdir -p .claude/commands

Step 2: Create .claude/commands/check-api-doc.md

Use the RTCCO template. Fill in YOUR context.

Step 3: Test it

/check-api-doc sample-project/test-docs/bad-api-doc.md

Step 4: Iterate

Pick one thing that didn't work. Fix it.

Common First-Round Fixes

ProblemFix
Role was too vagueMake it specific: "senior technical editor specializing in REST API documentation"
Output was unstructuredAdd an explicit table or list format
AI added too much commentaryAdd: "Output ONLY the table. No preamble."
Criteria were ambiguousDescribe what PASS looks like, not just the criterion name
False positivesNarrow the scope in Constraints
